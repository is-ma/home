=== CÓMO USAR ESTE TUTORIAL =========================================

Cada ejercicio crea un repositorio limpio en /tmp/lazygit-practice-*.
Los pasos de setup se indican al inicio de cada ejercicio.

IMPORTANTE:
  - LazyGit debe estar YA ABIERTO en cada ejercicio
  - No se incluye el paso de lanzamiento (asume lazygit ya ejecutado)
  - Cada ejercicio es independiente: siempre parte de repo vacío
  - Al terminar, limpia con: rm -rf /tmp/lg-practice-*

SETUP TÍPICO DE CADA EJERCICIO:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-X && \
  cd /tmp/lg-practice-X && git init && \
  git config user.name "Test" && git config user.email "test@test.com"

Reemplaza X con el número del ejercicio. Luego abre LazyGit en ese
directorio y practica los atajos indicados.


=== CONVENCIONES DE TECLAS ==========================================

  <Esc>        = Escape (cierra paneles, cancela operaciones)
  <Ret>        = Enter (confirma selección, ejecuta acción)
  <Tab>        = Tabulador (cambia entre paneles principales)
  <Space>      = Barra espaciadora (marca/desmarca archivos, staging)
  <C-c>        = Ctrl+C (cancela operación en curso, cierra popup)
  <C-z>        = Ctrl+Z (minimiza LazyGit temporalmente)
  <A-letra>    = Alt + letra (atajos compuestos)
  ?            = Abre menú de ayuda global
  /            = Abre búsqueda/filtro dentro del panel activo
  x            = Abre menú de contexto para selección actual


=== CONVENCIONES DE PANELES =========================================

Los atajos se especifican con el panel de contexto:

  Panel: atajo

Ejemplos:
  Files: d          = En panel Files, presiona d (stage/unstage)
  Commits: r        = En panel Commits, presiona r (revert commit)
  Stash: a          = En panel Stash, presiona a (apply stash)
  Branches: c       = En panel Branches, presiona c (create branch)
  Remotes: f        = En panel Remotes, presiona f (fetch)

Paneles principales (navegación con <Tab>):
  - Files: archivos modificados, staged, untracked
  - Commits: historial de commits locales
  - Branches: ramas locales y remotas
  - Stash: stashes guardados
  - Remotes: repositorios remotos
  - Reflog: historial de referencias


=== NOTA SOBRE LAZYGIT ==============================================

Versión: 0.40+
Bindings: por defecto (default keybindings)
Layout: default (paneles estándar)

Si tus atajos no coinciden, verifica tu configuración con:
  lazygit --version

Los ejercicios asumen la configuración por defecto de LazyGit 0.40+.
Si has personalizado keybindings en ~/.config/lazygit/config.yml,
algunos atajos pueden diferir.


=== CAPITULO 1: NAVEGACION Y PANELES ===

En LazyGit cada panel tiene su propio conjunto de atajos. Dominar la
navegacion entre paneles es la base para todo lo demas.


--- Rotar entre paneles principales con Tab --------------------------

ESCENARIO:
  LazyGit muestra multiples paneles (Files, Commits, Branches, Stash).
  Necesitas moverte rapidamente entre ellos para revisar diferentes
  aspectos del repositorio.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-1 && cd /tmp/lg-practice-1 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "A" > file1.txt && git add . && git commit -m "init"
  echo "B" > file2.txt && git add . && git commit -m "add file2"
  echo "C" > file3.txt

OBJETIVO:
  Navegar en ciclo: Files → Branches → Commits → Stash → Files

SOLUCION:
  Files: <Tab>         (va a Branches)
  Branches: <Tab>      (va a Commits)
  Commits: <Tab>       (va a Stash)
  Stash: <Tab>         (vuelve a Files)

VERIFICACION:
  git status         # debe mostrar file3.txt como untracked
  git log --oneline  # debe mostrar 2 commits (init, add file2)


--- Navegar items dentro de un panel con j/k -------------------------

ESCENARIO:
  El panel Commits tiene varios commits. Necesitas revisar el historial
  moviendote arriba y abajo para ver detalles de cada commit.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-2 && cd /tmp/lg-practice-2 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "1" > a.txt && git add . && git commit -m "commit 1"
  echo "2" > b.txt && git add . && git commit -m "commit 2"
  echo "3" > c.txt && git add . && git commit -m "commit 3"
  echo "4" > d.txt && git add . && git commit -m "commit 4"
  echo "5" > e.txt && git add . && git commit -m "commit 5"

OBJETIVO:
  En el panel Commits, navegar desde el commit mas reciente hasta el
  primero (commit 1) y volver al mas reciente.

SOLUCION:
  Files: <Tab>       (va a Branches)
  Branches: <Tab>    (va a Commits)
  Commits: j         (baja un commit)
  Commits: j         (baja otro commit)
  Commits: j         (baja otro commit)
  Commits: j         (baja otro commit)
  Commits: k         (sube un commit)
  Commits: k         (sube otro commit)
  Commits: k         (sube otro commit)
  Commits: k         (sube otro commit)
  Commits: k         (sube hasta el primer commit)

VERIFICACION:
  git status         # debe estar limpio, sin cambios pendientes
  git log --oneline  # debe mostrar 5 commits


--- Expandir/contraer secciones con espacio --------------------------

ESCENARIO:
  El panel Files muestra dos secciones: "Staged changes" y
  "Unstaged changes". El espacio expande o contrae cada seccion
  para mostrar u ocultar los archivos dentro de ella.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-3 && cd /tmp/lg-practice-3 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "staged1" > staged1.txt && echo "staged2" > staged2.txt
  echo "unstaged1" > unstaged1.txt && echo "unstaged2" > unstaged2.txt
  git add staged1.txt staged2.txt

OBJETIVO:
  En el panel Files, contraer la seccion "Staged changes" y luego
  expandirla nuevamente. Hacer lo mismo con "Unstaged changes".

SOLUCION:
  Files: <Space>     (contrae/expande "Staged changes")
  Files: <Space>     (vuelve a expandir "Staged changes")
  Files: j j         (baja a "Unstaged changes")
  Files: <Space>     (contrae/expande "Unstaged changes")
  Files: <Space>     (vuelve a expandir "Unstaged changes")

VERIFICACION:
  git status         # debe mostrar 2 staged y 2 unstaged files
  git log --oneline  # debe mostrar 0 commits (repo vacio)


--- Ir a primer/ultimo item y paginacion -----------------------------

ESCENARIO:
  Con muchos archivos en el panel Files, necesitas saltar rapidamente
  al inicio o final de la lista, o hacer scroll de media pagina.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-4 && cd /tmp/lg-practice-4 && git init && git config user.name "Test" && git config user.email "test@test.com"
  for i in $(seq 1 20); do echo "file $i" > file$i.txt; done

OBJETIVO:
  En el panel Files, ir al ultimo archivo con G, luego al primero con g,
  y finalmente hacer scroll de media pagina abajo y arriba.

SOLUCION:
  Files: G           (salta al ultimo item de la lista)
  Files: g           (salta al primer item de la lista)
  Files: <C-d>       (scroll media pagina abajo)
  Files: <C-u>       (scroll media pagina arriba)

VERIFICACION:
  git status         # debe mostrar 20 untracked files
  git log --oneline  # debe mostrar 0 commits (repo vacio)


--- Buscar dentro de un panel con / ----------------------------------

ESCENARIO:
  Tienes muchos archivos en el panel Files y necesitas encontrar
  rapidamente uno por nombre sin navegar manualmente por toda la lista.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-5 && cd /tmp/lg-practice-5 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "alpha" > alpha.txt && echo "beta" > beta.txt
  echo "gamma" > gamma.txt && echo "delta" > delta.txt
  echo "epsilon" > epsilon.txt

OBJETIVO:
  En el panel Files, usar la busqueda para encontrar el archivo
  "gamma.txt" rapidamente, luego salir del modo busqueda.

SOLUCION:
  Files: /           (abre filtro de busqueda)
  Files: gamma       (escribe el nombre a buscar)
  Files: <Esc>       (cierra el filtro de busqueda)
  Files: /           (abre filtro nuevamente)
  Files: epsilon     (busca otro archivo)
  Files: <Esc>       (cierra el filtro)

VERIFICACION:
    git status         # debe mostrar 5 untracked files
    git log --oneline  # debe mostrar 0 commits (repo vacio)


=== CAPITULO 2: STAGING ===

El staging area es el area de preparacion entre tu working directory y el
historial de commits. LazyGit te permite afinar exactamente QUE incluiras
en el siguiente commit, desde archivos completos hasta lineas individuales.


--- Stage un archivo nuevo con space ---------------------------------

ESCENARIO:
  Tienes archivos nuevos (untracked) y necesitas agregar solo uno al
  staging area, dejando los otros fuera del proximo commit.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-6 && cd /tmp/lg-practice-6 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "archivo1" > archivo1.txt
  echo "archivo2" > archivo2.txt
  echo "archivo3" > archivo3.txt

OBJETIVO:
  En el panel Files, hacer stage solo de "archivo2.txt" usando el atajo
  space, dejando archivo1.txt y archivo3.txt como untracked.

SOLUCION:
  Files: j             (baja a archivo2.txt)
  Files: <Space>       (stage archivo2.txt)

VERIFICACION:
  git status           # debe mostrar 1 staged file (archivo2.txt) y 2 untracked
  git diff --staged    # debe mostrar el contenido de archivo2.txt


--- Stage todos los archivos con a -----------------------------------

ESCENARIO:
  Tienes multiples archivos modificados y sin trackear. Quieres incluir
  todos en el staging area de una sola vez.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-7 && cd /tmp/lg-practice-7 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "uno" > uno.txt
  echo "dos" > dos.txt
  echo "tres" > tres.txt
  echo "cuatro" > cuatro.txt

OBJETIVO:
  En el panel Files, hacer stage de TODOS los archivos untracked de una
  sola vez usando el atajo global 'a'.

SOLUCION:
  Files: a             (stage todos los archivos del panel)

VERIFICACION:
  git status           # debe mostrar 4 staged files, 0 untracked
  git diff --staged    # debe mostrar los 4 archivos agregados


--- Unstage un archivo ya stageado con space -------------------------

ESCENARIO:
  Un archivo ya esta en el staging area pero decidiste que no debe ir
  en el proximo commit. Necesitas quitarlo del stage.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-8 && cd /tmp/lg-practice-8 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "staged1" > staged1.txt && echo "staged2" > staged2.txt && echo "staged3" > staged3.txt
  git add staged1.txt staged2.txt staged3.txt

OBJETIVO:
  En el panel Files, navegar a la seccion "Staged changes" y hacer
  unstage de "staged2.txt" usando space.

SOLUCION:
  Files: j             (baja a staged2.txt en la seccion Staged changes)
  Files: <Space>       (unstage archivo2.txt)

VERIFICACION:
  git status           # debe mostrar 2 staged files y 1 unstaged file
  git diff --staged    # debe mostrar solo staged1.txt y staged3.txt


--- Stage un hunk especifico con enter -------------------------------

ESCENARIO:
  Un archivo tiene dos cambios independientes (dos hunks). Solo quieres
  incluir el primer hunk en el commit, dejando el segundo fuera.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-9 && cd /tmp/lg-practice-9 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "linea1" > archivo.txt && echo "linea2" >> archivo.txt && echo "linea3" >> archivo.txt
  git add . && git commit -m "initial"
  echo "MODIFICACION 1" > temp.txt
  sed -i '1s/.*/MODIFICACION 1/' archivo.txt
  rm temp.txt
  echo "MODIFICACION 2" > temp.txt
  sed -i '3s/.*/MODIFICACION 2/' archivo.txt
  rm temp.txt

OBJETIVO:
  En el panel Files, entrar al modo diff con enter, navegar al primer
  hunk y hacer stage solo de ese hunk.

SOLUCION:
  Files: <Enter>       (abre el diff del archivo)
  Diff: <Tab>          (navega al primer hunk si es necesario)
  Diff: <Space>        (stage este hunk)
  Diff: <Esc>          (vuelve al panel Files)

VERIFICACION:
  git status           # debe mostrar 1 staged file con cambios parciales
  git diff --staged    # debe mostrar solo el primer hunk (linea 1)
  git diff             # debe mostrar el segundo hunk sin staged (linea 3)


--- Stage una linea especifica con enter + space ---------------------

ESCENARIO:
  Un archivo tiene varias lineas modificadas. Necesitas hacer stage de
  una sola linea especifica, no de todo el hunk.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-10 && cd /tmp/lg-practice-10 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "original1" > archivo.txt && echo "original2" >> archivo.txt && echo "original3" >> archivo.txt && echo "original4" >> archivo.txt
  git add . && git commit -m "initial"
  sed -i '1s/.*/CAMBIO1/' archivo.txt
  sed -i '2s/.*/CAMBIO2/' archivo.txt
  sed -i '3s/.*/CAMBIO3/' archivo.txt
  sed -i '4s/.*/CAMBIO4/' archivo.txt

OBJETIVO:
  En el panel Files, entrar al diff con enter, navegar a la linea 2 y
  hacer stage solo de esa linea usando space.

SOLUCION:
  Files: <Enter>       (abre el diff del archivo)
  Diff: j              (baja a la linea CAMBIO2)
  Diff: <Space>        (stage solo esta linea)
  Diff: <Esc>          (vuelve al panel Files)

VERIFICACION:
  git status           # debe mostrar 1 staged file
  git diff --staged    # debe mostrar solo la linea 2 (CAMBIO2)
  git diff             # debe mostrar lineas 1, 3, 4 sin staged


--- Unstage todo con U -----------------------------------------------

ESCENARIO:
  Varios archivos estan en el staging area pero decidiste reiniciar y
  quitar TODO del stage para empezar de cero.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-11 && cd /tmp/lg-practice-11 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "uno" > uno.txt && echo "dos" > dos.txt && echo "tres" > tres.txt
  git add uno.txt dos.txt tres.txt

OBJETIVO:
  En el panel Files, hacer unstage de TODOS los archivos de una vez
  usando el atajo 'U' (mayuscula).

SOLUCION:
  Files: U             (unstage todos los archivos)

VERIFICACION:
  git status           # debe mostrar 0 staged files, 3 unstaged files
  git diff --staged    # debe estar vacio (sin cambios staged)


--- Discard cambios de un archivo con d ------------------------------

ESCENARIO:
  Un archivo tiene modificaciones no deseadas en el working directory.
  Quieres descartar esos cambios y volver a la version del ultimo commit.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-12 && cd /tmp/lg-practice-12 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "original" > archivo.txt
  git add . && git commit -m "initial"
  echo "MODIFICACION NO DESEADA" > archivo.txt

OBJETIVO:
  En el panel Files, descartar los cambios del archivo modificado y
  restaurarlo a su estado original usando el atajo 'd'.

SOLUCION:
  Files: d             (abre menu de discard)
  Files: <Enter>       (confirma discard en el menu)

VERIFICACION:
  git status           # debe mostrar working tree clean
  cat archivo.txt      # debe mostrar "original"


--- Discard todos los cambios con D ----------------------------------

ESCENARIO:
  Varios archivos tienen modificaciones no deseadas. Quieres descartar
  TODOS los cambios de una vez y limpiar el working directory.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-13 && cd /tmp/lg-practice-13 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "uno" > uno.txt && echo "dos" > dos.txt && echo "tres" > tres.txt
  git add . && git commit -m "initial"
  echo "MOD1" > uno.txt
  echo "MOD2" > dos.txt
  echo "MOD3" > tres.txt

OBJETIVO:
  En el panel Files, descartar todos los cambios modificados de una vez
  usando el atajo 'D' (mayuscula).

SOLUCION:
  Files: D             (abre menu de discard all)
  Files: <Enter>       (confirma discard all en el menu)

VERIFICACION:
  git status           # debe mostrar working tree clean
  cat uno.txt          # debe mostrar "uno"
  cat dos.txt          # debe mostrar "dos"
  cat tres.txt         # debe mostrar "tres"


=== CAPITULO 3: COMMITS ===

El panel Commits muestra el historial de commits del branch actual. Desde aqui
puedes crear nuevos commits, modificar los existentes (amend, reword), deshacer
errores (undo, redo) y resetear el branch a estados anteriores (soft, mixed,
hard). Dominar estas operaciones te da control total sobre tu historial.


--- Commit con mensaje rapido Files: c -------------------------------

ESCENARIO:
  Tienes un archivo nuevo en el staging area y necesitas crear un commit
  con un mensaje corto de una linea.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-14 && cd /tmp/lg-practice-14 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "contenido nuevo" > archivo.txt
  git add archivo.txt

OBJETIVO:
  En el panel Files, con el archivo ya stageado, abrir el prompt de commit
  y crear el commit con un mensaje rapido usando 'c'.

SOLUCION:
  Files: c             (abre prompt de commit con mensaje)
  Files: "Mi commit"   (escribe el mensaje y presiona Enter)

VERIFICACION:
  git log --oneline    # debe mostrar 1 commit con mensaje "Mi commit"
  git status           # debe mostrar working tree clean


--- Commit con editor externo Files: C -------------------------------

ESCENARIO:
  Necesitas crear un commit con un mensaje largo y detallado que requiere
  edicion en tu editor de texto preferido (vim, nano, vscode).

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-15 && cd /tmp/lg-practice-15 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "feature importante" > feature.txt
  git add feature.txt

OBJETIVO:
  En el panel Files, abrir el editor externo para escribir un mensaje de
  commit multilinea usando 'C' (mayuscula).

SOLUCION:
  Files: C             (abre editor externo para commit)
  Editor: escribir mensaje multilinea y guardar

VERIFICACION:
  git log --oneline    # debe mostrar 1 commit
  git log -1           # debe mostrar el mensaje completo multilinea


--- Amend ultimo commit Commits: shift+A -----------------------------

ESCENARIO:
  Acabas de hacer un commit pero olvidaste incluir un archivo pequeno.
  Quieres agregar ese archivo al commit anterior sin crear uno nuevo.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-16 && cd /tmp/lg-practice-16 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "archivo1" > archivo1.txt
  git add archivo1.txt && git commit -m "primer commit"
  echo "archivo2" > archivo2.txt
  git add archivo2.txt

OBJETIVO:
  En el panel Commits, navegar al ultimo commit y hacer amend para incluir
  el archivo olvidado usando 'shift+A'.

SOLUCION:
  Commits: <Enter>     (navega al primer commit en la lista)
  Commits: shift+A     (amend ultimo commit con cambios staged)

VERIFICACION:
  git log --oneline    # debe mostrar 1 commit (no 2)
  git show --stat      # debe mostrar archivo1.txt Y archivo2.txt en el commit


--- Reword mensaje Commits: r ----------------------------------------

ESCENARIO:
  El ultimo commit tiene un mensaje incorrecto o poco descriptivo.
  Necesitas cambiar solo el mensaje sin modificar el contenido.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-17 && cd /tmp/lg-practice-17 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "contenido" > archivo.txt
  git add archivo.txt && git commit -m "mal mensaje"

OBJETIVO:
  En el panel Commits, seleccionar el ultimo commit y cambiar su mensaje
  usando 'r' (reword).

SOLUCION:
  Commits: <Enter>     (navega al commit)
  Commits: r           (abre prompt para reword)
  Commits: "mensaje corregido" (escribe nuevo mensaje)

VERIFICACION:
  git log --oneline    # debe mostrar 1 commit con mensaje "mensaje corregido"
  git log -1 --format=%s  # debe mostrar solo el nuevo mensaje


--- Undo ultimo commit Commits: z ------------------------------------

ESCENARIO:
  Acabas de hacer un commit pero te diste cuenta de que fue un error.
  Quieres deshacer ese commit y recuperar los cambios en el working directory.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-18 && cd /tmp/lg-practice-18 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "commit1" > archivo1.txt
  git add archivo1.txt && git commit -m "commit 1"
  echo "commit2" > archivo2.txt
  git add archivo2.txt && git commit -m "commit 2"

OBJETIVO:
  En el panel Commits, deshacer el ultimo commit (commit 2) y recuperar
  los cambios usando 'z'.

SOLUCION:
  Commits: z           (undo ultimo commit, cambios vuelven al working dir)

VERIFICACION:
  git log --oneline    # debe mostrar 1 commit (solo "commit 1")

--- Redo commit deshecho Commits: Ctrl+z -----------------------------

ESCENARIO:
  Hiciste undo de un commit pero te arrepentiste. Quieres recuperar el
  commit que acabas de deshacer.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-19 && cd /tmp/lg-practice-19 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "commit1" > archivo1.txt
  git add archivo1.txt && git commit -m "commit 1"
  echo "commit2" > archivo2.txt
  git add archivo2.txt && git commit -m "commit 2"
  # Simular que ya se hizo undo (archivo2.txt queda sin commitear)
  git reset --soft HEAD~1

OBJETIVO:
  En el panel Commits, recuperar el commit que fue deshecho con undo
  usando 'Ctrl+z' (redo).

SOLUCION:
  Commits: Ctrl+z      (redo del ultimo undo)

VERIFICACION:
  git log --oneline    # debe mostrar 2 commits nuevamente
  git status           # debe mostrar working tree clean


--- Reset soft Commits: g -> soft ------------------------------------

ESCENARIO:
  Tienes 3 commits y quieres deshacerlos TODOS, pero mantener los cambios

ESTADO INICIAL:

OBJETIVO:

  deshacer todos los commits manteniendo cambios en staging area.

SOLUCION:

visuales claros. Debes editar manualmente el archivo para resolver el conflicto,
marcarlo como resuelto y completar el merge. Aprenderas a identificar, resolver
y gestionar conflictos de merge de manera eficiente.

=== CAPITULO 6: REBASE INTERACTIVO ===

El rebase interactivo te permite reescribir el historial de commits de forma
controlada. Puedes reordenar, combinar (squash/fixup), editar, eliminar (drop)
o dividir commits. LazyGit abre un editor interactivo donde cambias la accion
de cada commit (pick, squash, fixup, reword, edit, drop, break, exec). Es
ideal para limpiar commits antes de hacer merge o preparar un PR profesional.
Advertencia: nunca hagas rebase de commits ya publicados en branches compartidas.


--- Reorder commits Commits: r ---------------------------------------

ESCENARIO:
  Tienes 4 commits en orden incorrecto: A, B, C, D. Quieres mover el commit C
  antes del commit B para que el historial quede en orden logico A, C, B, D.
  El rebase interactivo te permite reordenar commits arrastrandolos con j/k.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-* && mkdir /tmp/lg-practice-42 && cd /tmp/lg-practice-42 && git init && git config user.name "Test" && git config user.email "test@test.com"
  echo "A" > fileA.txt && git add . && git commit -m "feat: add A"
  echo "B" > fileB.txt && git add . && git commit -m "feat: add B"
  echo "C" > fileC.txt && git add . && git commit -m "feat: add C"
  echo "D" > fileD.txt && git add . && git commit -m "feat: add D"

OBJETIVO:
  Iniciar rebase interactivo desde el panel Commits, reordenar los commits
  moviendo C antes que B usando j/k para navegar y mover en la lista.

SOLUCION:
  Commits: r         (inicia rebase interactivo, abre editor con lista de commits)
  Commits: j j       (navega hasta el commit C)
  Commits: k         (mueve commit C una posicion hacia arriba, repite hasta estar antes de B)
  Commits: Ctrl+s    (guarda y ejecuta el rebase, o equivalente segun editor)
  # LazyGit aplica el rebase en orden

VERIFICACION:
  git log --oneline  # debe mostrar commits en orden: A, C, B, D
  ls                 # debe mostrar fileA.txt, fileC.txt, fileB.txt, fileD.txt

=== CAPITULO 11: REFLOG ===

El reflog es el registro de todas las referencias en tu repositorio local.
Cada vez que HEAD cambia (checkout, commit, reset, rebase), se guarda un entry.
Sirve para recuperar commits "perdidos", branches eliminadas y deshacer operaciones.

ESCENARIO:

  echo "v2" >> app.txt && git add . && git commit -m "commit 2"
  echo "v3" >> app.txt && git add . && git commit -m "commit 3"
  git checkout -b feature && echo "feature" >> app.txt && git add . && git commit -m "feature commit"
  git checkout main

OBJETIVO:
  Ver el reflog completo para inspeccionar todos los movimientos de HEAD
  que ocurrieron en el repositorio.

SOLUCION:

  Navegar: j/k              (explorar entries del reflog)
  Ver detalles: Enter       (muestra commit seleccionado)

VERIFICACION:

  # Debe mostrar: checkout main, feature commit, checkout feature, commit 3, commit 2, commit 1

--- Agregar submodulo desde repositorio local --------------------------

ESCENARIO:
  Tienes un proyecto principal que necesita incluir otro repositorio como submodulo
  por ejemplo una libreria compartida o configuracion compartida.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-*
  git config --global protocol.file.allow always
  mkdir -p /tmp/lg-practice-78-child && cd /tmp/lg-practice-78-child && git init
  git config user.name "Test" && git config user.email "test@test.com"
  echo "shared config" > config.txt && git add . && git commit -m "initial config"
  cd /tmp
  mkdir lg-practice-78-parent && cd lg-practice-78-parent && git init
  git config user.name "Test" && git config user.email "test@test.com"
  echo "main project" > main.txt && git add . && git commit -m "main project init"

OBJETIVO:
  Agregar el repositorio child como submodulo del repositorio parent
  y verificar que aparece en el panel de Submodules de LazyGit.

SOLUCION:
  Terminal: git submodule add /tmp/lg-practice-78-child lib-shared
  Terminal: git commit -m "add submodule"
  LazyGit: Panel numerico: 7    (abre panel de Submodules)
  # Ver el submodulo listado con su hash y path

VERIFICACION:

  ls -la                  # muestra directorio lib-shared
  cat .gitmodules         # muestra configuracion del submodulo

--- Actualizar submodulo a ultima version ------------------------------

ESCENARIO:
  El repositorio del submodulo ha recibido nuevos commits
  y necesitas actualizar tu submodulo a la ultima version.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-*
  git config --global protocol.file.allow always
  mkdir -p /tmp/lg-practice-79-child && cd /tmp/lg-practice-79-child && git init
  git config user.name "Test" && git config user.email "test@test.com"
  echo "v1" > lib.txt && git add . && git commit -m "lib v1"
  cd /tmp
  mkdir lg-practice-79-parent && cd lg-practice-79-parent && git init
  git config user.name "Test" && git config user.email "test@test.com"
  git submodule add /tmp/lg-practice-79-child lib-shared
  git commit -m "add submodule"
  cd lib-shared && echo "v2" >> lib.txt && git add . && git commit -m "lib v2"
  cd ..

OBJETIVO:
  Actualizar el submodulo a la ultima version del repositorio child
  usando LazyGit para ver el cambio pendiente.

SOLUCION:
  LazyGit: Panel numerico: 7        (abre panel de Submodules)
  # Ver submodulo con hash desactualizado (flecha o indicador)
  Seleccionar submodulo: Enter
  Update submodule: u               (update this submodule)
  # O actualizar todos: Shift+U     (update all submodules)
  Terminal: git add lib-shared
  Terminal: git commit -m "update submodule to v2"

VERIFICACION:
  git submodule status    # muestra nuevo hash del submodulo
  cat lib-shared/lib.txt  # muestra v1 y v2
  git log --oneline -3    # muestra commit de actualizacion

--- Sincronizar submodulo con remote -----------------------------------

ESCENARIO:
  El repositorio del submodulo cambio de ubicacion (nuevo remote URL)
  y necesitas sincronizar la configuracion del submodulo para que apunte
  a la nueva ubicacion sin tener que reinstalarlo.

ESTADO INICIAL:
  rm -rf /tmp/lg-practice-*
  git config --global protocol.file.allow always
  mkdir -p /tmp/lg-practice-80-child && cd /tmp/lg-practice-80-child && git init
  git config user.name "Test" && git config user.email "test@test.com"
  echo "lib content" > lib.txt && git add . && git commit -m "lib init"
  cd /tmp
  mkdir lg-practice-80-parent && cd lg-practice-80-parent && git init
  git config user.name "Test" && git config user.email "test@test.com"
  git submodule add /tmp/lg-practice-80-child lib-shared
  git commit -m "add submodule"
  # Simular cambio de remote
  cd lib-shared && git remote set-url origin https://github.com/example/lib-shared.git
  cd ..

OBJETIVO:
  Sincronizar la URL del remote del submodulo usando el panel de Submodules
  de LazyGit para actualizar la configuracion local.

SOLUCION:
  LazyGit: Panel numerico: 7        (abre panel de Submodules)
  Seleccionar submodulo: Enter
  Sync submodule: s                 (sync this submodule)
  # Actualiza la URL del remote en .git/config

VERIFICACION:
  git submodule status    # muestra estado del submodulo
  git config --get submodule.lib-shared.url  # muestra URL sincronizada
  cat .gitmodules         # muestra configuracion del submodulo
